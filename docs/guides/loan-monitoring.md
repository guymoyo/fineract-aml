# Post-Disbursement Loan Monitoring

This guide explains how the WeBank AML system monitors loan disbursements for post-disbursement fraud typologies and credit score gaming.

## 1. Overview

Every `LOAN_DISBURSEMENT` transaction detected by the webhook consumer triggers the creation of a `LoanDisbursementWatch` record. Two Celery tasks are then scheduled automatically:

- **+24h check**: Evaluates behavior in the first 24 hours after disbursement.
- **+48h check**: Evaluates cumulative behavior at the 48-hour mark.

Both checks run the same suite of behavioral tests. Alerts generated by loan monitoring use `source = "loan_monitoring"` to allow analysts to filter them separately from rule-engine or ML-sourced alerts.

### LoanDisbursementWatch Lifecycle

```
LOAN_DISBURSEMENT detected
    → LoanDisbursementWatch created (status: ACTIVE)
    → +24h check task scheduled
    → +48h check task scheduled

+24h check:
    → Suspicious behavior found → status: FLAGGED → Alert created
    → No suspicion yet          → status remains ACTIVE (await 48h)

+48h check:
    → Watch is FLAGGED          → skip (already handled)
    → Watch is ACTIVE (clean)   → status: CLEARED
```

Once a watch is `FLAGGED`, subsequent checks are skipped to avoid duplicate alerts. A watch reaches `CLEARED` only if both check windows pass without suspicious activity.

## 2. Behavior Checks

The following checks are evaluated at each scheduled check point (+24h and +48h):

### Loan-and-Run

**Trigger**: More than 80% of the disbursed amount has been transferred out within 24 hours of disbursement.

**Severity**: 0.90

This is the primary loan fraud typology for mobile lending — the borrower has no intention of repaying and immediately liquidates the loan. The 80% threshold accounts for legitimate partial utilization while catching rapid full extraction.

### Immediate Cash-Out

**Trigger**: A withdrawal exceeding 80% of the loan amount occurs within 30 minutes of disbursement.

**Severity**: 0.95

The highest-severity loan monitoring rule. A withdrawal of this size within 30 minutes indicates the loan was obtained solely to convert to cash, typically indicating a stolen-identity or synthetic-identity loan. The 30-minute window distinguishes this from legitimate same-day utilization.

### Post-Disbursement Structuring

**Trigger**: 3 or more sub-threshold transfers whose amounts sum to approximately the loan disbursement amount, occurring after disbursement.

**Severity**: 0.75

The borrower splits the loan into sub-CTR-threshold transfers to avoid detection. The "approximately equal" sum check uses a tolerance of ±10% to account for transfer fees.

### Cross-Agent Dispersal

**Trigger**: Loan funds sent to 5 or more distinct counterparties within 24 hours of disbursement.

**Severity**: 0.70

Rapid dispersal to many counterparties is associated with mule network seeding — the loan proceeds are distributed across a network of controlled accounts before being aggregated elsewhere. Lower severity than other rules because some legitimate use cases (e.g., paying multiple suppliers) can produce similar patterns.

## 3. Alert Source

All alerts generated by loan monitoring are tagged with:

```
source = "loan_monitoring"
```

This allows compliance analysts to:
- Filter the alert queue to loan-monitoring-specific alerts
- Report separately on post-disbursement fraud rates
- Exclude loan monitoring alerts from rule-engine or ML accuracy metrics

## 4. Credit Score Gaming Detection

Before generating a credit recommendation, the system checks for signs that a customer has artificially inflated their apparent transaction volume to obtain a higher credit limit.

**Detection condition**: The 7-day inbound transaction total is greater than 3× the average weekly inbound total over the prior 30 days.

The credit scoring feature vector also includes a **`round_trip_score`** feature (defined in `app/features/credit_extractor.py`). It measures the fraction of inbound funds that left to the same counterparty within 48 hours, flagging wash transactions that inflate apparent turnover without representing genuine economic activity:

```
round_trip_score = round_trip_amount / total_inflow   (capped at 1.0)
```

A high `round_trip_score` provides an independent ML signal for gaming, complementing the rule-based 3× multiplier check.

**When detected:**

| Effect | Value |
|--------|-------|
| `score_inflation_flag` | `True` |
| Score penalty | −0.15 applied to the raw credit score |
| Recommendation override | Forced to `REVIEW_CAREFULLY` regardless of underlying score |
| Reviewer notes | Inflation warning included with the ratio that triggered the flag |

The 3× multiplier is calibrated to avoid penalizing genuinely positive income events (e.g., a salary increase, a one-time bonus) while catching customers who have coordinated rapid deposits in the days before applying for credit.

## 5. LLM Credit Explanation

When `AML_LLM_INVESTIGATION_ENABLED=true`, each new credit request triggers a call to the Claude API to generate a dual-audience explanation. The explanation is stored in `CreditRequest.explanation_text`.

### Customer Version (French)

A plain-language explanation of the credit score result written in French, targeted at the customer. Includes:
- The primary factors that raised or lowered the score
- Concrete improvement tips the customer can act on
- A reassuring or constructive tone appropriate for customer communication

### Compliance Version

A technical explanation targeted at compliance reviewers. Includes:
- Key scoring factors with their weights and values
- Whether the `score_inflation_flag` was set and the inflation ratio
- The recommendation rationale, including any overrides applied

Both versions are stored together in `explanation_text` as a structured JSON object with `customer_fr` and `compliance_en` keys.
